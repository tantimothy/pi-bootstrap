# 🐳 Raspberry Pi Multi-Environment Docker Deployer

A smart, stateless, and lightweight TUI (Terminal User Interface) deployment dashboard for managing multi-environment Docker configurations on a Raspberry Pi. 

This repository allows you to skip hosting large Docker images in external registries. Instead, it pulls raw code or environment workspaces directly from GitHub and compiles them locally using the Raspberry Pi's native ARM processor.

---

## 🏗️ Repository Structure

Organize your project into isolated workspace folders inside the `environments/` directory. Each folder can contain its own independent setup assets, localized metadata configuration templates (`.env.example`), or custom hardware-access profiles.

```text
├── deploy.sh                 # The main deployment orchestrator script
├── README.md                 # This documentation file
└── environments/             # Isolated app workspaces
    ├── dev-compose/          # Setup Type 1: Docker Compose
    │   ├── .env.example      # Interactive secret template layout
    │   └── docker-compose.yml
    ├── prod-custom-run/      # Setup Type 2: Custom Bash Run Execution
    │   ├── .env.example
    │   ├── Dockerfile
    │   └── run.sh
    └── standalone-app/       # Setup Type 3: Pure Dockerfile Fallback
        └── Dockerfile
```

### The Universal `deploy.sh` Script
This version scans for all subdirectories inside `environments/`. Once you pick one, it automatically manages dependencies, pulls structural variables, manages container teardowns, and handles execution paths:
1. Is there an uncommitted local `.env` file? If missing and a `.env.example` blueprint exists, parse comments as TUI forms to safely collect secret inputs.
2. Is there a `run.sh`? If yes, run it.
3. Is there a `docker-compose.yml`? If yes, run `docker compose up`.
4. Is there just a `Dockerfile`? If yes, dynamically generate a standard docker run command for it.

### Writing a Custom `run.sh`

For folders where a simple `Dockerfile` isn't enough and you don't want to use Compose, you can write a `run.sh` script to contain complex CLI execution parameters. To ensure it runs perfectly before user permission groups refresh, consume the exported `$DOCKER_CMD` engine alias.

Example `environments/prod-custom-run/run.sh`:

```bash
#!/bin/bash
DOCKER=${DOCKER_CMD:-docker}

# Source local dynamic secrets if generated by the TUI wizard
if [ -f ".env" ]; then
    export $(cat .env | xargs)
fi

echo "📦 Compiling custom local build..."
$DOCKER build --no-cache -t custom-app:latest .

echo "🏃 Running container with specialized flags..."
$DOCKER run -d \
  --name=my_custom_container \
  --restart=unless-stopped \
  -p 8080:3000 \
  -v /home/pi/logs:/app/logs \
  --device /dev/gpiomem \
  -e APP_SECRET="$MY_SECRET_KEY" \
  custom-app:latest
```

_(Notice how you can even pass through Raspberry Pi hardware access parameters like `--device /dev/gpiomem` directly via the custom script flag options!)_

## 🚀 Interactive Deployment Modes

You can run the deployment routine in two flexible configurations:

### Mode A: Stateless Execution (Zero-Setup Remote Curl)
Perfect for brand new or stateless Pis. You do not even need to copy the deployment script manually. Run this single command to pull down, sync, and launch the deployment suite. 

> ⚠️ **Security Note:** Prefixing the line with your inline environment variables keeps your Personal Access Token (PAT) secure and prevents it from being logged in your shell's `.bash_history` file.

```bash
CURL_USER="tantimothy:<your_github_token>" curl -sSL --user "tantimothy:<your_github_token>" \
https://raw.githubusercontent.com/tantimothy/pi-bootstrap/main/deploy.sh | bash
```
*(The orchestration engine decodes this header token internally and automatically injects it into Git runtime headers using `git -c http.extraHeader`. This ensures that downstream operations like `git clone` or `git reset` bypass interactive username/password terminal credential prompts entirely.)*

### Mode B: Local Repository Execution
If you clone this repository down to your Pi to test edits manually, the script automatically detects it is inside a valid git working directory. It will bypass generic fallback paths, fetch upstream branches, and cleanly align your active path against the origin remote:

```bash
CURL_USER="tantimothy:<your_github_token>" ./deploy.sh
```

## 🔐 Zero-Commit Interactive Secret Dashboard

If an environment requires local parameters (such as API keys, database hashes, or hardware paths), save a `.env.example` file directly into that subfolder. Make sure to add `.env` to your global repository `.gitignore`!

The deployment engine features a **Custom Metadata Syntax Pre-Processor** that interprets comment blocks (`#`) as user-friendly explanations and values following `=` as default form placeholders:

### 1. Write the Workspace Blueprint (`.env.example`)
```ini
# The core operational API key allocated from your Shodan dashboard engine.
SHODAN_API_KEY=default_public_token_here

# The interface identifier matching your local system monitoring hardware array.
WIFI_INTERFACE=wlan0

# The security fallback credentials used to lock down the administrative routing control portal.
ADMIN_PASSWORD=
```

### 2. Streamlined Multi-Field UI Generation
When a workspace is selected for deployment:
1. **The Legend Overlay Screen:** It displays a full-screen scrollable dialog window documenting precisely what each variable does based on your template comments.
2. **The Consolidated Form Dashboard:** It serves a single unified entry layout pre-populated with your defined defaults. Arrow keys swap fields, and inputs are saved securely to a local, uncommitted `.env` file on the Pi filesystem.
3. **Graceful Aborts:** Pressing `Cancel` during the configuration wizard deletes the working draft and safely halts the pipeline before hitting container compilation layers.

## 🛠️ Supported Environment Types

When an environment workspace is selected from the menu interface, the orchestration engine evaluates files sequentially and selects the best matching routine:

1. **Custom Shell (`run.sh`):** Best for advanced options (e.g., matching Raspberry Pi hardware access arrays like `--device /dev/gpiomem`).
2. **Docker Compose (`docker-compose.yml`):** Best for multi-container microservice stacks. Compose automatically reads the local generated `.env` variables.
3. **Pure Dockerfile (`Dockerfile`):** Universal automated build fallback. Maps internal app services automatically to port `80` and automatically passes values using `--env-file .env`.
